最严重的是任务队列并不适合多实例/多 worker 生产运行。app/core/application.py (line 55) 会在应用启动时直接拉起 worker，而 app/workflows/task_queue.py (line 184) 只是先查一条 queued 再改成 running，没有 FOR UPDATE SKIP LOCKED 或原子更新。单机还能凑合，多进程/多实例下有重复消费风险。
数据库层处于“重构半收口”状态。app.db 当前只导出 get_db_session 和 initialize_database，见 app/db/init.py#L1 (line 1)，但测试还在直接 import create_connection，见 tests/test_settings_api.py (line 1) 和 tests/test_project_api.py (line 8)。我实际跑这两个测试都在收集阶段直接报 ImportError。这说明目录结构已经改了，但公共接口和测试没完全同步。
schema 的“唯一真相”不够清晰。app/alembic/env.py (line 22) 明显在走 ORM + Alembic 元数据，但 app/db/base.py (line 5) 又写着建表仍由 app/db/mysql.py (line 1) 里的手写 SQL 负责。生产里最怕这种“双轨制”，因为迁移、回滚、初始化很容易漂移。
服务层有明显的“巨石化”问题。app/services/project_service.py (line 43) 这个文件我统计下来大约 5005 行，已经不是正常单服务体量了。它更像把项目、工作流、外部视频模型接入、消息、分镜、素材处理都揉在一起了。能跑，但后续改动、排障、加人协作都会越来越难。
同步/异步边界不够干净。比如 app/api/routers/projects.py (line 15) 是 async def 路由，但里面直接调用同步数据库服务；而 app/services/project_service.py (line 43)、app/services/asset_service.py (line 14)、app/services/job_service.py (line 11)、app/services/system_settings_service.py (line 16) 都是手动 next(get_db_session()) 拿同步 Session。这在低并发可以接受，但在生产下会让 event loop 被阻塞，性能和稳定性都会受影响。
目录本身还有过渡态痕迹。repositories/ 已经建出来了，但我全局查下来，主干业务几乎没真正用上；同时还留着空目录 app/database、app/processors、app/crawlers/douyin。这说明架构方向有了，但还没完全收口。

作为“稳定、多人协作、可扩容、可回滚、可持续维护的生产级后端”，还不够。
