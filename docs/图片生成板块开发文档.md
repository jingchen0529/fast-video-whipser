# 图片生成板块开发文档

## 1. 文档目的

本文档是图片生成板块的主开发文档，描述图片生成板块自身的前端展示、后端接口、工作流执行、系统模型配置和资产库落库方案。

本文档只覆盖图片生成，不展开视频分析、动作资产提取和视频复刻板块。

### 范围说明

1. 本文档覆盖图片生成板块。
2. 图片生成板块独立于 `motion_extraction`、`remake`。
3. 最终生成结果统一落到 `media_assets`，在前端“资产库”中可见。

---

## 2. 当前仓库现状

### 2.1 已有能力

当前仓库已经具备以下可复用能力：

1. `projects` 主线任务流已可承接异步项目执行。
2. `WorkflowRegistry / WorkflowEngine / TaskQueue` 已具备工作流执行基础。
3. `media_assets` 已可保存 `image` 类型资产。
4. `AssetService.create_asset()` 已能创建图片资产记录。
5. 前端已有：
   - 工作台模式切换基础
   - 历史详情页轮询基础
   - 设置页 Provider 配置页基础

### 2.2 当前缺口

图片生成板块目前没有专门实现，缺口主要有：

1. 没有 `image` 相关 `workflow_type`。
2. 没有图片生成工作流定义。
3. 系统设置中还没有独立的 `image` 模型分组。
4. 前端没有图片生成入口和结果展示协议。
5. 图片生成结果虽然能落 `media_assets`，但没有标准回写结构。

---

## 3. 板块目标

### 3.1 业务目标

支持以下两类使用方式：

1. 文生图
   - 用户输入提示词
   - 系统调用图片模型生成结果
2. 参考图改写
   - 用户上传参考图
   - 系统保留主体或风格方向进行再生成

### 3.2 技术目标

统一沿用现有项目工作流模式：

1. 创建任务走 `POST /api/projects/upload`
2. 详情查询走 `GET /api/projects/{id}`
3. 结果元数据写入 `projects.image_generation_json` 或扩展 `video_generation_json` 的通用生成结构
4. 最终图片文件落库到 `media_assets`

说明：

从字段设计角度，更推荐新增 `image_generation_json`，不要继续把图片结果塞进 `video_generation_json`。

---

## 4. 推荐的数据结构

### 4.1 新增字段建议

建议在 `projects` 表新增：

1. `image_generation_json TEXT`

默认结构建议为：

```json
{
  "status": "idle",
  "provider": null,
  "model": null,
  "objective": null,
  "prompt": null,
  "negative_prompt": null,
  "reference_asset_ids": [],
  "provider_task_id": null,
  "output_asset_ids": [],
  "result_images": [],
  "error_detail": null,
  "updated_at": null
}
```

### 4.2 `media_assets` 落库约定

图片生成成功后，每张图都应创建一条 `media_assets` 记录：

1. `asset_type=image`
2. `source_type=generated`
3. `file_name`
4. `file_path`
5. `mime_type`
6. `size_bytes`
7. `width`
8. `height`
9. `metadata_json`

## 6. 推荐工作流

### 6.1 工作流类型

建议新增：

1. `workflow_type=image`

### 6.2 工作流步骤

建议定义 `ImageGenerationWorkflow`，共 6 步：

1. `define_objective`
   - 解析用户生成目标
2. `collect_references`
   - 收集参考图、风格图、品牌素材
3. `build_image_prompt`
   - 构造图片模型 Prompt
4. `generate_images`
   - 调用图片模型
5. `save_results`
   - 下载结果并落库 `media_assets`
6. `finish`
   - 更新项目状态为 `succeeded`

### 6.3 步骤说明

#### Step 1: `define_objective`

输入：

1. 用户输入的 prompt
2. 可选的图片用途说明

输出：

1. 主题
2. 风格
3. 主体
4. 尺寸方向

#### Step 2: `collect_references`

支持：

1. 上传图片
2. 从资产库中选择已有图片
3. 无参考图时直接跳过

#### Step 3: `build_image_prompt`

输出建议包含：

1. `prompt`
2. `negative_prompt`
3. `aspect_ratio`
4. `image_count`
5. `style_preset`

#### Step 4: `generate_images`

读取：

1. `settings.image.default_provider`
2. `settings.image.providers`

输出：

1. `provider`
2. `model`
3. `provider_task_id`

#### Step 5: `save_results`

必须做的事：

1. 下载第三方生成结果
2. 为每张图创建 `media_assets`
3. 回写 `output_asset_ids`
4. 回写图片 URL 列表

#### Step 6: `finish`

项目状态必须更新为：

1. `succeeded`
2. 失败时为 `failed`

## 7. 接口设计

### 7.1 创建任务

建议继续复用：

```http
POST /api/projects/upload
Content-Type: multipart/form-data
```

表单字段：

1. `workflow_type=image`
2. `objective`
3. `file` 或 `reference_asset_ids`

说明：

如果当前接口不方便传 `reference_asset_ids`，可以先只支持单张上传参考图，后续再扩展。

### 7.2 查询详情

```http
GET /api/projects/{id}
```

返回中应包含：

1. `workflow_type`
2. `status`
3. `task_steps`
4. `image_generation`

### 7.3 资产库查询

继续复用：

```http
GET /api/assets?asset_type=image
```

## 10. 最小可用版本

MVP 建议先做到：

1. 仅支持文生图
2. 仅支持一个 Provider
3. 一次生成 1-4 张
4. 生成成功后立即落库到 `media_assets`
5. 前端能查看结果并进入资产库

---

## 11. 验收标准

满足以下条件即算图片生成板块可用：

1. 前端可以创建 `workflow_type=image` 的项目
2. 后端能从 `settings.image` 读取默认模型配置
3. 图片生成工作流能正常推进并结束
4. 生成结果能写入 `image_generation_json`
5. 生成结果能落库到 `media_assets`
6. 前端能展示图片结果并在资产库看到对应记录

## 13. 实施顺序建议

建议按下面顺序落地：

1. 先补 `settings.image`
2. 再补 `workflow_type=image`
3. 再补图片生成结果落库
4. 最后做前端工作台和历史页展示

这样可以先打通后端最小链路，再补前端体验。
