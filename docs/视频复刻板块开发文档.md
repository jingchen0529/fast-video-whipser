# 视频复刻板块开发文档

## 1. 文档目的

本文档是视频复刻板块的唯一主开发文档。该板块独立于 `motion_extraction`，只描述视频复刻自身的前端展示、后端接口、工作流执行器、系统视频模型调用和动态资产落库。

本文档只描述基于当前仓库实现可直接落地的方案，不引入新的架构层级。

### 范围说明

1. 本文档覆盖 `remake` 板块。
2. 视频分析板块已视为前置能力，不在本文展开。
3. 动作资产板块不是本板块前置条件，如后续引用，按独立来源处理。

---

## 2. 现状结论

### 2.1 已有能力

当前仓库已经具备以下基础能力：

1. 前端工作台已经支持三种模式切换：
   - `analysis`：脚本分析
   - `remake`：复刻爆款
   - `create`：创作爆款
2. 前端新建任务统一走 `POST /api/projects/upload`
3. 后端 `projects.workflow_type` 已支持：
   - `analysis`
   - `remake`
   - `create`
4. 工作流引擎已注册：
   - `AnalysisWorkflow`
   - `RemakeWorkflow`
   - `CreateWorkflow`
5. 系统设置已经有视频模型分组：
   - `GET /api/settings`
   - `PATCH /api/settings`
   - `settings.remake.default_provider`
   - `settings.remake.providers[*]`
6. 项目详情已经预留 `video_generation` 字段用于展示视频生成结果
7. 仓库中已经存在 `media_assets` 资产表，可承接复刻结果入库

### 2.2 当前阻塞点

虽然骨架已存在，但“视频复刻 / 爆款复刻”板块还没有真正可执行，主要阻塞如下：

1. `app/services/project_service.py` 中 `_step_remake_*` 仍是占位实现，只返回文案，没有真实调用视频模型。
2. `remake` 工作流没有最终 `finish` 步骤，项目状态不会被更新为 `succeeded`。
3. `create` 工作流的结束状态被写成 `completed`，但前端轮询终态判断只认 `succeeded / failed / ready`，会导致页面可能一直轮询。
4. 前端虽然能创建 `workflow_type=remake` 的项目，但没有复刻结果的专门展示协议，实际仍依赖通用 `video_generation` 字段。
5. 当前“复刻爆款”按钮本质上只是把 `workflow_type=remake` 传给后端，并未细分“普通视频复刻”和“爆款复刻”两类输入语义。
6. 当前文档中的生成结果示例仍偏向“返回 URL”，但业务要求是最终必须落库到动态资产，不能只停留在第三方返回地址。

## 3. 目标定义

### 3.1 业务目标

补齐两类任务流：

1. 视频复刻
   - 用户提供一个参考视频或上传本地视频
   - 系统提取参考信息
   - 调用系统配置中的视频模型生成复刻版本
   - 结果回写项目详情并在前端可查看
2. 爆款复刻
   - 用户提供一个爆款视频链接或上传爆款素材
   - 系统保留核心节奏、镜头语言和卖点结构
   - 系统按照用户目标进行改写
   - 调用同一套视频模型生成结果

### 3.2 技术目标

统一使用现有项目工作流主线，不新增新的任务系统：

1. 创建任务：`POST /api/projects/upload`
2. 查询详情：`GET /api/projects/{id}`
3. 轮询执行状态：复用现有 `history.vue` / `workbench.vue`
4. 模型配置来源：`settings.remake`
5. 结果展示载体：`projects.video_generation_json`
6. 最终视频资产落库到 `media_assets`，并在前端“动态资产”中可见

---

## 4. 术语映射

为避免命名混乱，本次建议统一如下：

1. `workflow_type=remake`
   - 统称：复刻工作流
   - 对外文案可显示为“视频复刻 / 爆款复刻”
2. `workflow_type=create`
   - 统称：创作工作流
   - 不属于本文重点
3. “视频复刻”和“爆款复刻”
   - 都走 `remake` 工作流
   - 区别由 `objective` 文本和参考素材来源决定

如果后续一定要拆成两个后端工作流，建议新增子类型字段，而不是再扩散新的 `workflow_type`。

## 5. 推荐的复刻任务流

### 5.1 工作流定义

建议将 `remake` 工作流收敛为 7 步：

1. `select_source_asset`
   - 识别参考来源
   - 支持上传视频、本地素材、外链视频
   - 输出：`source_asset_id`、`source_url`、`reference_frames`
2. `analyze_reference_video`
   - 提取参考视频的结构化特征
   - 输出：节奏、镜头描述、主体、场景、风格、卖点线索
3. `define_remake_intent`
   - 解析用户希望“保留什么 / 改写什么”
   - 输出：保留动作、保留节奏、改写人物、改写场景、改写文案等
4. `build_remake_prompt`
   - 基于参考分析结果和用户意图生成视频模型输入
   - 输出：`prompt`、`negative_prompt`、`duration`、`aspect_ratio`
5. `generate_video`
   - 调用系统设置中的视频模型
   - 输出：`provider`、`model`、`provider_task_id`
6. `poll_generation_result`
   - 轮询第三方任务结果
   - 输出：`result_video_url`、`cover_url`、`error_detail`
7. `finish`
   - 将复刻结果落库到动态资产，并更新项目状态为 `succeeded`

### 5.2 为什么要加 `analyze_reference_video` 和 `finish`

1. 当前 `_step_remake_select_source` 到 `_step_remake_save_result` 过于粗糙，缺少真正的参考视频理解。
2. 当前 `remake` 没有终态收口步骤，前端无法稳定识别任务已完成。
3. 参考视频分析应该复用已有分析链路的部分能力，而不是直接拼文案。

## 6. 爆款复刻的语义规则

“爆款复刻”不建议单独新开工作流，建议作为 `remake` 的一种任务意图。

### 6.1 输入规则

当前接口继续保持：

```http
POST /api/projects/upload
Content-Type: multipart/form-data
```

表单字段：

1. `workflow_type=remake`
2. `objective`
3. `source_url` 或 `file`

### 6.2 objective 约定

前端在“复刻爆款”模式下，建议把输入组织成结构化文本，例如：

```text
任务类型：爆款复刻
保留项：节奏、镜头切换、卖点结构
改写项：人物、场景、产品文案
目标平台：TikTok
目标人群：北美女性 25-35 岁
商品卖点：防水、轻便、便携
风格偏好：真实口播 + 生活化场景
```

后端先按文本协议解析；如果解析失败，再回退为普通自然语言。

### 6.3 爆款复刻与普通视频复刻的差异

1. 普通视频复刻
   - 更强调画面、动作、运镜、色彩
2. 爆款复刻
   - 更强调节奏、卖点结构、转化链路、镜头编排

两者都可复用同一视频生成模型配置，但构造的 prompt 不同。

## 7. 系统配置调用规范

视频模型必须来自系统设置中的 `remake` 分组，禁止在代码里硬编码视频模型。

### 7.1 读取规则

后端执行 `generate_video` 时必须：

1. 读取 `SystemSettingsService().get_settings()`
2. 取 `settings["remake"]["default_provider"]`
3. 在 `settings["remake"]["providers"]` 中找到对应 provider 配置
4. 使用以下字段发起视频生成任务：
   - `provider`
   - `base_url`
   - `api_key`
   - `default_model`

### 7.2 落库规则

复刻视频最终必须落库到动态资产，不允许只保存第三方临时 URL。

这里的“动态资产”后端落点统一为 `media_assets` 表；如果前端菜单名称仍显示为“动态资产”，则以该名称对外展示即可。

建议落库顺序：

1. 第三方视频生成成功
2. 后端下载生成结果到本地受管目录
3. 在 `media_assets` 中创建一条 `asset_type=video`、`source_type=generated` 记录
4. 将该资产 ID 和访问 URL 回写到 `projects.video_generation_json`
5. 如果本次复刻是基于某个已有参考素材或源视频派生，补充资产血缘关系

建议最少写入以下字段：

1. `file_name`
2. `file_path`
3. `mime_type`
4. `size_bytes`
5. `duration_ms`
6. `width`
7. `height`
8. `thumbnail_path`

### 7.3 回写规则

每次调用后必须把关键执行信息写回 `projects.video_generation_json`：

```json
{
  "status": "running",
  "provider": "doubao",
  "model": "seedance-pro",
  "objective": "爆款复刻",
  "asset_type": "video",
  "asset_name": null,
  "asset_url": null,
  "prompt": "...",
  "provider_task_id": "xxx",
  "result_video_url": null,
  "error_detail": null,
  "updated_at": "2026-04-10T00:00:00Z"
}
```

任务完成后：

```json
{
  "status": "succeeded",
  "provider": "doubao",
  "model": "seedance-pro",
  "objective": "爆款复刻",
  "asset_type": "video",
  "asset_name": "viral-remake-20260410.mp4",
  "asset_url": "/api/uploads/generated/viral-remake-20260410.mp4",
  "prompt": "...",
  "provider_task_id": "xxx",
  "result_video_url": "/uploads/generated/xxx.mp4",
  "error_detail": null,
  "updated_at": "2026-04-10T00:00:00Z"
}
```

失败时：

```json
{
  "status": "failed",
  "provider": "doubao",
  "model": "seedance-pro",
  "asset_type": "video",
  "asset_name": null,
  "asset_url": null,
  "provider_task_id": "xxx",
  "result_video_url": null,
  "error_detail": "第三方视频生成失败",
  "updated_at": "2026-04-10T00:00:00Z"
}
```

## 8. 前端接入要求

### 8.1 工作台

当前 `frontend/app/pages/video/workbench.vue` 已经具备基本接入能力：

1. 通过模式切换把 `remake` 传给 `workflow_type`
2. 调用 `/projects/upload`
3. 创建成功后跳转 `/video/history?id={project.id}`

前端需要补齐的不是新接口，而是展示规则。

### 8.2 推荐展示规则

在 `history.vue` 中增加复刻结果展示区，读取 `project.video_generation`：

1. `status=running`
   - 展示“视频生成中”
   - 显示 provider / model / provider_task_id
2. `status=succeeded`
   - 展示生成视频播放器
   - 展示 prompt 摘要
   - 展示参考视频与生成结果对比
   - 提供“查看动态资产”入口，跳转或联动到对应资产
3. `status=failed`
   - 展示失败原因

### 8.3 时间线图标补齐

当前 `TaskTimeline.vue` 只对分析流程的 step key 做了图标映射。

需要补齐：

1. `select_source_asset`
2. `analyze_reference_video`
3. `define_remake_intent`
4. `build_remake_prompt`
5. `generate_video`
6. `poll_generation_result`
7. `finish`

否则复刻流程会显示通用图标，信息密度不够。

### 8.4 页面文案修正

建议修正以下文案，避免误导：

1. `workbench.vue` 页面标题现在是“视频分析工作台”，应改为更通用的“视频工作台”
2. `history.vue` 中如果项目类型为 `remake`，顶部标题应显示“视频复刻详情”或“爆款复刻详情”

## 9. 后端实现要求

### 9.1 `ProjectService` 必须补齐的逻辑

#### A. 复刻工作流最终状态

当前问题：

1. `remake` 没有 `finish` 步骤
2. `create` 结束时写入的是 `completed`

必须调整为：

1. `remake` 新增 `finish` 步骤，并在最后调用：

```python
self._update_project(
    project_id=project_id,
    status="succeeded",
    summary="视频复刻工作流已完成。",
    error_message=None,
)
```

2. `create` 的终态也统一为 `succeeded`

#### B. 视频模型调用逻辑

`_step_remake_generate_video()` 不能再返回占位文案，至少要完成以下动作：

1. 读取 `settings.remake`
2. 选中默认 provider
3. 构造模型请求参数
4. 发起生成任务
5. 把任务信息回写 `video_generation_json`

#### C. 结果轮询逻辑

建议新增 `_step_remake_poll_result()`：

1. 根据 `provider_task_id` 查询第三方状态
2. 成功时下载结果文件并落库到 `media_assets`
3. 失败时抛出异常，让工作流进入 `failed`

#### D. 动态资产入库逻辑

复刻成功后必须新增一段统一入库逻辑，建议由 `AssetService` 承接：

1. 将第三方生成视频保存为受管文件
2. 创建 `media_assets` 记录，标记为生成资产
3. 将 `media_asset.id` 写入 `video_generation` 扩展字段，例如 `output_asset_id`
4. 将 `asset_url` 写为该资产的公开访问地址
5. 如存在上游参考素材，记录派生关系，保证后续可追溯

### 9.2 输出字段规范

`GET /api/projects/{id}` 返回的 `video_generation` 至少要包含：

1. `status`
2. `provider`
3. `model`
4. `objective`
5. `asset_type`
6. `asset_name`
7. `asset_url`
8. `output_asset_id`
9. `prompt`
10. `provider_task_id`
11. `result_video_url`
12. `error_detail`
13. `updated_at`

### 9.3 消息流规范

建议保留现有消息流风格，在关键步骤写入 assistant 消息：

1. `workflow_status`
2. `workflow_error`

对于生成成功结果，可追加一条结构化消息：

1. `message_type=video_generation_result`
2. `content=复刻视频已生成完成`
3. `content_json` 中带 `result_video_url`

## 11. 验收标准

### 11.1 功能验收

满足以下条件即算视频复刻板块可用：

1. 前端选择“复刻爆款”后可以成功创建 `workflow_type=remake` 项目
2. 后端能从 `settings.remake` 读取默认视频模型
3. 工作流能把每一步状态写入 `project_task_steps`
4. 任务结束后项目状态为 `succeeded` 或 `failed`
5. `GET /api/projects/{id}` 能返回完整 `video_generation`
6. 复刻生成的视频已落库到 `media_assets`
7. 前端历史页能展示视频生成结果，并能关联到动态资产

### 11.2 最小可用版本

MVP 建议先做到：

1. 单一 provider 打通，例如 `doubao`
2. 只支持一个参考视频
3. 只支持文生视频或图生视频中的一种
4. 生成成功后必须立即下载并入库到动态资产，不保留“只存远程 URL 不入库”的实现

## 13. 实施顺序建议

建议按以下顺序落地：

1. 先修正 `remake/create` 工作流终态，保证前端不会卡在轮询中
2. 再补 `video_generation_json` 的写入协议
3. 然后接真实视频模型调用
4. 最后补前端结果展示和复刻细节文案

这样可以先打通最小可用版本，再逐步增强模型能力。
